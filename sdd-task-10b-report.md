## UPC-Dedup Logic

### POST (create)
1. If `data.upc` is present: look up by `{ upc }` via `dbQuery().findOne`. If found → return existing row at HTTP 200 (dedup: one shared global row). If not found → create with `creator: me`, `visibility: 'public'`, emit 201.
2. If `data.upc` is absent: create with `creator: me`, `visibility: data.visibility ?? 'private'` (private by default). Emit 201.
3. `creator` is always stamped server-side; any client-supplied `creator` field in body is destructured away and ignored.

### PUT (update/upsert)
1. If `data.upc` present: check for existing row by `{ upc }` via `dbQuery().findOne`. If found → return existing row (convergence; client's recordId yields to shared row). If not found → create new row with `recordId` from URL param, `creator: me`, `visibility: 'public'`.
2. If `data.upc` absent: activity-style upsert by recordId+creator. Look up `{ recordId, creator: { id: me } }` → update if found. If recordId claimed by another creator → 404. Otherwise create with `creator: me`, `visibility: data.visibility ?? 'private'`.

## Concerns

1. **`as any` UID cast in controller and routes:** The `strapi build` process compiles TS before regenerating `types/generated/contentTypes.d.ts`, so a new UID is unknown to the type system on the first build. The workaround (`const UID = '...' as any`) is consistent with how the project would handle any new content type before the types are generated by `strapi develop`. After the first `strapi develop` run the types will be regenerated with the new UID registered, at which point the cast could be removed. The existing consumable controller uses `as const` because it was already in the generated types.

2. **`servingsPerContainer` as `decimal` stored as string in Postgres:** The `sanitizeFacts` utility coerces decimal→number for facts fields; however `servingsPerContainer` is a top-level attribute (not inside facts), so it is NOT coerced by `sanitizeFacts`. If running against Postgres (compose), `servingsPerContainer` may be returned as a string rather than a number. The test only runs against SQLite so this passes, but if a caller does `ConsumableProduct.fromJSON(strapiRowToShape(row))` and `servingsPerContainer` arrives as a string, `fromJSON` would silently ignore it (it checks `typeof servingsPerContainer === 'number'`). This is a known Postgres decimal-as-string quirk that could be addressed in a follow-up.

3. **`netWeight` stored as JSON:** JSON columns are returned as parsed objects by both SQLite and Postgres — no coercion needed. This is fine.
